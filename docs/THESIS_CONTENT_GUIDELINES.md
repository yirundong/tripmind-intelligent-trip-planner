# 毕业论文内容写作注意事项（仿照上届优秀论文）

本文档只关注“内容怎么写”，不重复排版细则。  
目标：在借鉴优秀论文写法的前提下，写出**与你自己的项目强绑定**、可通过答辩追问的论文。

## 1. 总原则：仿照“方法”，不仿照“内容”

- 可以仿照：章节组织方式、论证顺序、图表类型、问题分析套路、测试呈现方式。
- 不能照搬：研究背景表述、技术细节、图表数据、结论措辞、案例描述。
- 每一章都要能回答一句话：**“这部分在我的项目里具体做了什么？”**
- 避免空话套话（如“具有重要意义”“提高效率”），必须给出场景、指标或证据。

## 2. 按章节写作的核心要求

## 2.1 第1章 概述

- 选题背景：要落到真实业务场景（例如行程规划痛点：多目标、实时性、预算约束）。
- 国内外研究现状：按“国内/国外”分开写，且有近5年代表文献支撑。
- 相关技术：只写你项目实际用到的技术，不堆技术名词。
- 本文工作：用条目化方式写“本论文完成了什么”，和后续章节形成一一对应。

## 2.2 第2章 需求分析

- 从“用户角色”出发写需求，不从代码模块出发。
- 功能需求要对应具体用例（谁、在什么场景、完成什么目标）。
- 非功能需求要可验证（性能指标、可用性目标、稳定性指标）。
- 建议为每类需求增加“验收标准”，后续测试章可直接复用。

## 2.3 第3章 系统设计

- 先总体设计思想，再展开模块设计（不要直接贴模块图）。
- 数据库设计要体现推导过程：业务对象 -> 实体关系 -> 逻辑结构 -> 表结构。
- 每个模块设计都写清：
  - 输入是什么；
  - 核心处理逻辑是什么；
  - 输出是什么；
  - 与其它模块如何交互。
- 若涉及算法，写“为什么选它 + 怎么落地到你的业务”。

## 2.4 第4章 实现与测试

- 实现章节写“实现思路与关键代码机制”，不是“软件操作手册”。
- 按模块说明页面、类、方法职责，避免大段贴代码。
- 测试章节必须有：测试方法、测试环境、测试用例、结果分析。
- 用例要覆盖正常流程 + 边界条件 + 异常流程（成功与失败都要有）。
- 测试结果不能只写“通过”，要写指标与现象（例如响应时间、命中率、错误率）。

## 2.5 总结与展望

- 总结写“做成了什么 + 证据是什么 + 局限是什么”。
- 展望要结合当前缺陷，给出可执行改进方向，不写空泛愿景。

## 3. 文献与引用注意事项

- 文献不要只堆数量，要和正文观点有对应关系（每条引用都“有用”）。
- 优先选近5年文献，核心算法/基础理论可保留经典老文献。
- 中外文献比例要满足学院要求（通常要求英文文献占一定比例）。
- 引用格式统一，正文引用与参考文献条目必须一一对应。
- 不要出现“参考文献中有、正文没引用”或“正文引用了、文末没条目”。

## 4. 图、表、流程图的内容原则

- 每张图表都要回答问题：这张图证明了什么？
- 图前有引导语，图后有结论语，不能只插图不解释。
- 流程图/时序图要体现你项目特有元素（业务对象、模块命名、数据表名）。
- 表格数据要有来源与含义说明，避免“只列数据不分析”。
- 同类图表命名与编号规则保持一致。

## 5. 面向“智能行程规划”项目的重点提醒

- 不要只写“做了个系统”，要突出“规划智能性”：
  - 约束条件（时间、预算、偏好、交通方式）如何建模；
  - 推荐或搜索策略如何权衡；
  - 冲突条件如何处理（例如预算不足、时间冲突、地点过远）。
- 需求和测试中必须体现真实用户任务：
  - 快速生成路线；
  - 修改偏好后二次重规划；
  - 异常场景（数据缺失、接口超时）下系统行为。
- 若使用第三方接口/地图服务，要交代依赖边界、失败降级策略和缓存策略。

## 6. 老师高频关注点（答辩与审阅风险）

- 是否“紧扣自己题目”还是模板化套话。
- 是否有“分析过程”而不是仅给结果图。
- 是否体现工程思维：需求 -> 设计 -> 实现 -> 测试的闭环。
- 是否能解释关键技术选型的理由与替代方案。
- 是否能证明系统“有效”，而不是“能跑起来”。

## 7. 常见扣分点（避免）

- 章节之间断裂：前文提到的目标，后文没有实现或验证。
- 只写功能，不写约束、边界、失败处理。
- 测试用例全是成功路径，没有异常和边界。
- 贴大量代码、截图堆砌，但缺少文字分析。
- 用词绝对化（“完全解决”“显著提升”）却没有数据支撑。
- 图表与正文脱节，编号混乱，引用不到位。

## 8. 写作执行流程（建议）

- 第一步：先列“章节 -> 子节 -> 每节一句目标”。
- 第二步：先写第2/3/4章（你最熟悉的需求、设计、实现），再补第1章综述。
- 第三步：整理测试证据（日志、截图、结果表）并回填到第4章。
- 第四步：根据正文反推摘要与关键词，最后写总结与展望。
- 第五步：做一次“交叉一致性检查”（章节承诺、图表编号、文献引用、术语统一）。

## 9. 提交前自查清单（内容向）

- [ ] 每章是否明确回答“做了什么、为什么这样做、效果如何”  
- [ ] 国内外研究现状是否分开写且有近期文献支撑  
- [ ] 需求分析是否按用户视角，而不是按代码模块罗列  
- [ ] 设计章节是否包含设计过程，不只是结果图  
- [ ] 实现章节是否避免操作手册化与代码堆砌  
- [ ] 测试是否覆盖成功、失败、边界，并有结果分析  
- [ ] 总结是否包含成果、局限、可落地改进方向  
- [ ] 全文是否能体现“这是你的项目，而非模板替换”
