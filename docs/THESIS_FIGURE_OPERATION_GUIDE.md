# 论文配图制作操作手册（对应 `THESIS_DRAFT_INTEGRATED.md`）

本手册用于指导你把论文中的“留空图位”快速补齐，重点保证**规范、统一、可答辩解释**。  
建议工具组合：`draw.io`（流程/架构图）+ `Navicat/pgAdmin`（数据库图）+ `项目实际截图`（界面图）+ `Word题注与交叉引用`（编号管理）。

## 1. 总体原则

- 图必须服务论证，不是装饰：每张图都回答一个问题。
- 图名与正文描述一致：正文先引用“如图X-X所示”，再给图。
- 命名统一：`图2-1`、`图3-4` 这种“章号-序号”格式。
- 样式统一：字体、线条粗细、主色调、箭头风格保持一致。
- 导出统一：优先PNG，宽度建议 1600px 以上，避免糊图。

## 2. 每张图怎么做（逐图说明）

## 2.1 图2-1 系统用例图

**工具**：draw.io / ProcessOn / StarUML  
**步骤**：
1. 画两个角色：普通用户、管理员。  
2. 画系统边界框“智能旅行规划系统”。  
3. 在边界框内放用例椭圆：注册登录、提交规划、查看结果、管理行程、收藏地点、后台管理等。  
4. 用连线连接角色与可执行用例。  
5. 检查：普通用户不要连后台管理；管理员不要连普通用户私人操作。  
6. 导出PNG，命名为 `fig_2_1_usecase.png`。

## 2.2 图2-2 核心功能活动图

**工具**：draw.io  
**步骤**：
1. 起点 -> 输入需求 -> 参数校验。  
2. 分支：校验通过/失败。  
3. 通过分支进入“创建规划任务 -> 多Agent执行 -> 生成结果 -> 展示与保存”。  
4. 失败分支返回“提示修正输入”。  
5. 结束节点闭合。  
6. 导出 `fig_2_2_activity.png`。

## 2.3 图3-1 系统总体架构图

**工具**：draw.io / Figma  
**图层建议**：
- 第一层：前端层（Vue3页面、路由、状态管理）。  
- 第二层：后端层（API网关、业务服务、智能体调度）。  
- 第三层：智能体层（景点Agent、天气Agent、酒店Agent、规划Agent、校验Agent）。  
- 第四层：数据与外部层（PostgreSQL、日志、地图API）。  
**关键**：箭头要表达“请求流”和“数据流”方向。

## 2.4 图3-2 实体关系概念图（E-R图）

**工具**：draw.io / MySQL Workbench / pgModeler  
**步骤**：
1. 列实体：User、TripPlan、TripItem、Favorite、Place、PlanTask、AuditLog。  
2. 标主键（PK）和外键（FK）。  
3. 标关系基数：1:N、N:1。  
4. 只保留核心字段（不要塞太细，细节放表结构表格）。  
5. 导出 `fig_3_2_er.png`。

## 2.5 图3-3 核心数据表关系图

**工具**：Navicat / DBeaver / pgAdmin  
**步骤**：
1. 在本地数据库建好表结构。  
2. 用ER图工具自动生成关系图。  
3. 调整布局，保证线不交叉过多。  
4. 截图后适当裁剪，导出高清PNG。  
5. 导出 `fig_3_3_tables.png`。

## 2.6 图3-4 用户认证时序图

**工具**：draw.io / PlantUML  
**参与者**：用户、前端、认证接口、数据库、JWT服务。  
**主流程**：登录请求 -> 参数校验 -> 用户验证 -> 签发Token -> 返回前端 -> 后续接口携带Token。  
**异常流程**：密码错误/Token过期。  
导出 `fig_3_4_auth_seq.png`。

## 2.7 图3-5 多智能体工作流图

**工具**：draw.io  
**建议结构**：
- Start  
- 输入标准化节点  
- 并行扇出节点（景点/天气/酒店）  
- 聚合节点  
- 规划生成节点  
- 校验节点  
- 条件分支（通过/修复）  
- End  
导出 `fig_3_5_agent_flow.png`。

## 2.8 图3-6 校验与修复流程图

**重点**：体现“校验失败 -> 修复重试 -> 达到上限 -> 兜底返回”的闭环。  
导出 `fig_3_6_repair.png`。

## 2.9 图4-1 项目目录结构图

**做法A（推荐）**：在IDE展开目录后截图。  
**做法B**：命令行树形输出后截图。  
注意隐藏无关目录（如依赖缓存、临时文件）。  
导出 `fig_4_1_project_tree.png`。

## 2.10 图4-2 前后端交互时序图

**参与者**：浏览器、前端、后端API、Agent服务、数据库、外部地图API。  
至少包含两条链路：  
- 链路1：登录鉴权；  
- 链路2：提交规划到结果返回。  
导出 `fig_4_2_fe_be_seq.png`。

## 2.11 图4-3 到 图4-9 页面截图类图

包含：
- 登录页  
- 规划结果页  
- 行程管理页  
- 收藏页  
- 地图标注页  
- 后台管理页  

**截图规范**：
1. 浏览器缩放 100%。  
2. 分辨率建议 1920x1080。  
3. 每页保留核心区域，去掉无关隐私信息。  
4. 命名示例：`fig_4_5_plan_result.png`。  
5. 必要时用浅色矩形遮盖敏感数据。

## 3. 表格如何制作（对应表3-1~表4-3）

- 表3-1~3-3：来自数据库字段设计，列建议为“字段名/类型/是否为空/说明”。  
- 表4-1~4-3：测试表建议为“用例ID/场景/输入/预期/实际/结论”。  
- 在Word中使用统一表格样式，表头加粗居中，正文小五或小四按模板要求。

## 4. Word中插图与题注操作（关键）

1. 插入图片：`插入 -> 图片 -> 本地文件`。  
2. 题注：右键图片 `插入题注`，标签选“图”，编号格式设置为“包含章节号”。  
3. 题注格式统一为“图X-X 图名”。  
4. 在正文引用：`引用 -> 交叉引用 -> 仅标签和编号`，避免手输编号。  
5. 若新增或删除图片，按 `Ctrl+A -> F9` 全文更新编号与目录。

## 5. 图目录与检查清单

## 5.1 生成图目录

- 使用Word“引用 -> 插入图表目录”，自动生成图目录。
- 目录更新同样使用 `Ctrl+A -> F9`。

## 5.2 提交前检查

- [ ] 图号连续，无跳号重号  
- [ ] 正文均有“如图X-X所示”引用  
- [ ] 图清晰无马赛克、无遮挡关键信息  
- [ ] 图题表述准确，不出现“截图1”“图片2”这类非学术命名  
- [ ] 图与章节内容强相关，可被答辩解释  

## 6. 建议的出图顺序（省时间）

1. 先画结构图（图3-1、图3-5、图3-6）  
2. 再画需求图（图2-1、图2-2）  
3. 然后导数据库图（图3-2、图3-3）  
4. 最后批量截图实现类图（图4-1~图4-9）  

这样做能保证你先把“论文骨架图”补齐，后续页面截图可按开发进度逐步替换。
