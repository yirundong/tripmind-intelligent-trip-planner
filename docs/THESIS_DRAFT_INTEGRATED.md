# 摘 要

随着居民消费升级与出行场景多样化，用户对旅游服务的需求已经从“信息检索”转向“智能决策与自动生成”。传统在线旅游平台虽然能够提供景点、酒店、交通等信息，但在跨平台数据整合、复杂约束协同计算、动态行程调整和个性化推荐方面仍存在明显不足。用户在实际规划过程中通常需要在多个网站和应用之间重复搜索、手动比对，面临信息分散、决策成本高、试错代价大、方案可执行性不稳定等问题。针对上述痛点，本文围绕“基于AI-Agent的智能旅行规划”展开研究与工程实现，构建了一个面向真实用户场景的全栈Web应用系统，并重点探索多智能体协同机制在旅行规划任务中的可行性和工程落地方法。

本文系统采用前后端分离架构：前端使用Vue3与TypeScript构建交互界面，后端使用FastAPI提供REST接口服务，数据库采用PostgreSQL并结合SQLAlchemy进行持久化管理，认证机制使用JWT实现用户身份校验与权限控制。智能规划能力方面，系统基于LangGraph状态图工作流构建多Agent协同引擎，将一次完整的旅行规划任务拆解为景点信息采集、天气信息采集、酒店推荐、行程编排与结果校验等子任务，通过并行与顺序混合协同策略实现“高可用、可解释、可扩展”的规划流程。系统在工具调用层接入地图与POI相关外部服务，使模型输出建立在可验证的事实数据基础上，降低纯生成模式下的幻觉风险；在结构化输出层采用Pydantic模型对结果进行校验和修复，提升方案的一致性和可落地性。

在需求分析层面，本文从用户角色与业务流程出发，系统梳理了功能需求、非功能需求与可行性约束；在系统设计层面，围绕架构分层、数据库建模、模块职责划分和接口规范展开详细论证；在实现与测试层面，完成用户认证、智能规划、行程管理、收藏探索与后台管理等核心模块开发，并通过功能测试、边界测试、异常测试和结果分析验证系统有效性。实验与测试结果表明，该系统能够在多数常见输入条件下稳定返回结构化旅行方案，支持用户对行程进行查看、收藏、再编辑与历史管理，具备较好的可用性与扩展潜力。

本文研究工作的工程价值在于：给出了一条从LLM能力到垂直场景智能应用的完整实现路径，证明了多Agent协同机制在旅行规划任务中的实际应用价值；研究价值在于：通过“需求—设计—实现—测试”闭环过程，形成了可复用的智能规划系统开发范式。未来可在多目标优化、实时重规划、个性化记忆建模和评估指标体系方面继续深化，以进一步提升系统的智能水平与工业可用性。

# Abstract

With the continuous growth of personalized travel demand, users now expect travel systems to provide not only information retrieval but also intelligent decision-making and itinerary generation. Traditional online travel platforms can provide points of interest, hotel data, and route suggestions, yet they still rely heavily on manual comparison and fragmented information browsing. This thesis proposes and implements an AI-Agent-based intelligent travel planning web application, aiming to provide an end-to-end solution for personalized itinerary generation under practical constraints.

The system adopts a front-end and back-end separated architecture. The front end is implemented with Vue3 and TypeScript, while the backend is built on FastAPI with PostgreSQL and SQLAlchemy for data persistence. JWT is used for authentication and access control. For intelligent planning, the system introduces a LangGraph-based multi-agent workflow that decomposes the planning task into specialized subtasks including attraction retrieval, weather query, hotel recommendation, itinerary composition, and output verification. External tools are integrated to provide factual constraints, and Pydantic schema validation is used to enforce structured output quality.

This thesis presents requirement analysis, architecture design, database modeling, module-level implementation, and comprehensive testing. Core modules including user management, itinerary planning, itinerary management, favorites and exploration, and administration are implemented and verified. Test results show that the system can generate stable and structured travel plans for major user scenarios, with acceptable interaction performance and functional reliability.

The contribution of this work lies in providing a practical engineering paradigm for applying LLM-driven multi-agent collaboration to a vertical domain. It also demonstrates that intelligent itinerary planning can be systematically developed through a complete lifecycle of requirement-design-implementation-testing. Future work may focus on multi-objective optimization, dynamic replanning, user preference memory, and fine-grained evaluation metrics.

Key words: AI-Agent; Multi-Agent Collaboration; LangGraph; Intelligent Travel Planning; FastAPI; Vue3; PostgreSQL

# 第1章 概 述

## 1.1 选题背景和意义

随着数字经济发展和旅游消费升级，用户对于旅行规划服务的要求发生了根本变化。过去用户主要依赖攻略文章、社交平台与搜索引擎自行拼装行程，当前则更希望系统能够“理解需求并直接给出可执行方案”。特别是在短期游、跨城游与自由行场景下，用户通常同时关注目的地吸引力、预算约束、天气变化、交通便利度、住宿条件与个性偏好，决策维度多且冲突关系复杂。传统工具大多只解决“查信息”的问题，没有真正解决“做决策”的问题。

在实际旅行规划流程中，常见问题包括：第一，信息源高度分散，景点、酒店、天气、路线数据分属不同平台，用户需要重复检索；第二，约束条件难以统一建模，例如“预算有限但希望覆盖多个核心景点”往往需要反复手动调整；第三，方案缺乏动态修正能力，当天气变化、景点关闭或交通拥堵时，用户需要从头规划；第四，平台推荐逻辑不透明，用户难以理解推荐依据，导致信任成本上升。上述问题说明，旅行规划场景天然适合引入具备任务分解、工具调用和结果整合能力的智能体系统。

大语言模型的快速发展为该问题提供了新的技术路径。与传统规则系统相比，LLM具备更强的自然语言理解与语义生成能力；与单轮问答式AI相比，AI-Agent能够通过规划—执行—观察—反思闭环处理多步骤任务。将多个专业化Agent组织为协同系统后，可把复杂任务拆解为可控子任务，再由统一调度层完成信息汇聚和方案输出，显著提升复杂业务场景下的处理能力。

本课题的工程意义主要体现在三个方面：其一，验证多智能体协同在旅行规划场景中的可行性，形成可复用的工程模板；其二，打通“用户需求输入—工具事实查询—结构化方案生成—前端可视化展示”的闭环链路，提升系统可落地程度；其三，在全栈项目中实践现代软件工程方法，构建具备完整业务流程与扩展接口的网络应用。研究意义在于探索LLM垂直应用的系统化实现方式，为后续相关课题提供可参考的需求建模、架构设计与验证思路。

## 1.2 国内外研究现状

### 1.2.1 国外研究现状

国外研究在大语言模型、Agent框架和智能推荐系统三条主线推进较快。大模型方面，GPT系列、Llama系列等模型在自然语言理解、代码生成和复杂推理任务中持续迭代，为Agent应用提供了底层能力。方法层面，Chain-of-Thought、ReAct、Toolformer等思路推动模型从“文本生成器”向“可执行智能体”演进，尤其是ReAct将推理与行动交替结合，增强了模型在多步骤任务中的可解释性与鲁棒性。

在Agent工程框架方面，LangChain提供了工具抽象、记忆管理和提示词模板等基础能力；LangGraph进一步通过状态图机制支持并行节点、条件分支与循环修复，适配复杂流程编排需求。AutoGPT、CrewAI、MetaGPT等项目则从不同角度尝试多角色协同与自动任务分解。旅行服务领域中，国外OTA平台已引入AI问答和推荐功能，但多数仍停留在文本建议或单点推荐层，真正将多源事实数据与结构化行程编排深度融合的系统仍相对有限。

近年来也有研究关注“事实约束下的生成式规划”。相关工作强调将外部API、检索系统或知识库接入模型推理流程，以降低幻觉并提升结果可执行性。该方向与本文采用的“工具调用+结构化校验”路线一致，即先获取事实数据，再进行方案生成与验证，避免纯生成导致的不确定输出。

### 1.2.2 国内研究现状

国内在大模型基础能力建设与行业落地方面同样发展迅速。文心、通义、智谱等模型持续更新，推动多场景智能应用开发。旅游行业内，携程、飞猪、同程等平台逐步引入AI助手用于问答和路线建议，但多数产品仍偏重“内容推荐+检索增强”，在多Agent协同、复杂约束处理和结构化输出方面尚有提升空间。

学术研究方面，国内对多智能体系统、推荐系统、路径规划与约束优化已有较深积累，但将LLM驱动Agent协同直接嵌入全流程Web系统的案例仍较少。特别是在本科毕业设计或中小规模应用场景中，如何以可控成本完成“模型、工具、业务、前后端”的一体化实现，仍缺少标准化范式。本文基于此背景，尝试提供一套可执行、可复用的工程方案。

### 1.2.3 评述与问题归纳

综合国内外研究可见，现有工作虽在模型能力和单项技术上进展显著，但在旅行规划场景仍存在四类问题：一是对真实业务约束考虑不足，结果可执行性较弱；二是多智能体协同机制描述较多、工程实现较少；三是系统评测多集中于模型指标，缺乏完整产品链路验证；四是输出格式不稳定，难以直接用于前端展示和后续管理。本文围绕上述问题，提出“多Agent协同+工具事实约束+结构化输出校验+全栈可运行系统”的综合实现路径。

## 1.3 研究内容

本文研究内容可归纳为以下六个方面：

第一，面向旅行规划业务建立需求模型。围绕用户从“输入需求”到“获得方案”再到“管理行程”的全流程，梳理核心用例、业务规则和系统边界。

第二，构建多智能体协同工作流。将复杂规划任务拆分为信息采集、推荐、编排、校验等子任务，定义角色职责和数据传递格式，形成可维护的任务链。

第三，设计并实现可扩展的系统架构。以前后端分离为基础，建立接口层、服务层、智能体层、数据层协同机制，保证系统可扩展性与可测试性。

第四，建立结构化数据模型与校验机制。通过Pydantic与数据库模型双重约束，确保输出结果能够稳定落地到前端展示和历史记录管理。

第五，完成核心模块开发与集成。实现用户认证、规划生成、行程管理、收藏探索和管理后台等模块，并完成接口联调与异常处理。

第六，构建测试与评估体系。设计功能测试、边界测试和异常测试用例，验证系统在典型输入下的稳定性、正确性和可用性。

## 1.4 相关技术介绍

### 1.4.1 大语言模型与AI-Agent

大语言模型具备强语言建模能力，但若直接用于复杂业务，容易出现“看似合理但缺乏事实依据”的输出。AI-Agent通过引入任务分解、工具调用和结果反思机制，使模型从“文本生成”升级为“任务执行”。在本系统中，Agent不仅负责生成文案，更承担调用外部服务、整合结构化数据和遵循业务规则等责任，是系统智能能力的核心载体。

### 1.4.2 LangGraph与LangChain

LangChain提供模型与工具连接的基础设施，LangGraph提供面向工作流的状态图执行机制。二者结合后，系统可实现节点级并行执行、条件分支、重试修复与状态持久化，更适合实现旅行规划这类“多步骤+多数据源+多约束”的任务。与单链式调用相比，状态图方案在可调试性和稳定性方面更具优势。

### 1.4.3 FastAPI、Pydantic与异步机制

FastAPI具备高性能、类型友好和自动文档等优势，适合快速构建规范化后端服务。Pydantic用于请求与响应校验，可在接口层拦截非法输入，提升系统健壮性。对于耗时规划任务，系统采用异步处理与状态轮询机制，避免接口长时间阻塞，提升前端交互体验。

### 1.4.4 Vue3与TypeScript

前端基于Vue3组合式API与TypeScript静态类型系统，形成“组件化+类型化”开发模式。通过路由守卫、状态管理和复用组件，系统实现登录鉴权、规划输入、结果展示、历史管理和后台管理等页面功能，能够在复杂交互下保持可维护性。

### 1.4.5 PostgreSQL与SQLAlchemy

PostgreSQL提供稳定可靠的数据存储能力，支持复杂查询与事务控制。SQLAlchemy作为ORM层，降低了业务代码与数据库耦合度，便于模型迁移和版本迭代。本文围绕用户、行程、收藏、会话与审计等对象建立数据表，为系统运行与后续扩展提供数据基础。

### 1.4.6 JWT认证与权限控制

系统采用JWT进行身份认证，通过Access Token与Refresh Token机制兼顾安全性与体验。在权限层面区分普通用户与管理员角色，对高风险接口执行权限校验和审计记录，降低越权访问风险。

# 第2章 系统需求分析

## 2.1 可行性分析

### 2.1.1 技术可行性

从技术栈成熟度看，本系统选用的Vue3、FastAPI、PostgreSQL、JWT、SQLAlchemy均为行业广泛使用技术，社区文档完善，工程风险可控。多智能体编排框架LangGraph已具备稳定版本与可观测调试能力，可支持状态流转、并行执行与异常分支，满足旅行规划复杂流程需求。外部服务方面，地图和POI接口具备标准化调用方式，能够提供景点位置、地理距离、评分与周边信息，为规划生成提供可靠事实基础。

从算法可行性看，旅行规划任务可拆解为若干独立子任务：景点筛选、天气判断、酒店推荐、路线组织、预算评估。通过“子任务并行采集+规划Agent整合生成+校验Agent检查修复”模式，可显著降低一次性大提示词生成失败概率。系统还引入备用生成逻辑，确保在结构化解析失败时返回基础可用方案，技术风险处于可接受范围。

### 2.1.2 操作可行性

系统面向普通用户设计，交互流程遵循“低学习成本”原则：用户只需输入目的地、出行日期、预算和偏好等核心参数，即可触发规划任务；结果页采用卡片化与时间线方式展示，便于快速浏览。对于管理员，后台界面提供用户管理、内容审核、日志查看等功能，操作路径清晰。整体交互不依赖专业知识，操作可行性高。

### 2.1.3 经济可行性

项目以开源框架为主，开发成本主要来自时间投入与必要的接口调用费用。部署可采用单机或轻量云服务器，硬件成本可控。系统可在毕业设计阶段以原型方式运行，不需要高并发商业级基础设施，因此经济可行性较好。后续若扩展为生产系统，可按用户规模逐步升级资源。

### 2.1.4 时间可行性

按照毕业设计周期，项目分为需求分析、架构设计、功能开发、联调测试、论文撰写五阶段。由于核心技术栈已有实践基础，且模块边界较清晰，可并行推进前后端开发与测试准备工作。通过里程碑管理（原型可用、核心功能完成、测试闭环完成）可保障在规定时间内交付。

### 2.1.5 环境可行性

开发环境可在常规Windows/Linux设备上完成，依赖安装明确。运行环境支持本地部署与云端部署，数据库、后端、前端均可容器化管理。系统对硬件要求适中，网络环境要求为可访问外部工具接口，环境约束可满足。

### 3.2.2 数据库逻辑设计

行程管理模块支持行程保存、查看、编辑、删除、复制与再次规划。系统采用“行程主表+日程子表”结构，实现数据组织清晰化。编辑行为支持局部修改（如替换某日酒店）和全局修改（如整体预算调整）。删除操作采用软删除策略，保留审计信息。复制功能可把历史方案作为新请求模板，降低重复输入成本。

### 3.3.4 收藏与探索模块设计

收藏模块支持景点/酒店一键收藏、分类查看和快速加入行程。探索模块根据城市、主题、评分、热度进行筛选展示。为提升体验，系统在前端维护本地状态并结合后端分页接口，减少重复请求。收藏数据既可作为用户主动管理入口，也可为后续偏好学习提供基础。

### 3.3.5 管理后台模块设计

后台模块面向管理员，提供用户管理、内容治理、系统监控与日志审计。设计重点在于权限隔离与操作可追溯：所有敏感操作写入审计日志；重要配置修改需二次确认；异常任务支持按状态、时间、用户维度过滤查询。后台设计使系统具备基本运维能力，为后续上线部署奠定基础。

### 3.3.6 接口规范与错误处理设计

系统接口统一采用REST风格，返回结构包含状态码、消息、数据与追踪ID。错误处理分为参数错误、鉴权错误、业务错误、外部服务错误和系统错误五类，对应不同错误码与提示信息。对于可恢复错误（如外部查询超时），系统优先返回可解释提示并允许重试；对于不可恢复错误，系统返回统一兜底消息并记录详细日志。

## 3.4 关键算法与协同策略设计

### 3.4.1 任务分解策略

任务分解遵循“高内聚、低耦合”原则。景点、天气、酒店信息采集任务相互独立，适合并行执行；规划与校验任务依赖上游结果，采用顺序执行。该策略既利用并行能力降低等待时间，又保证最终方案的一致性。

### 3.4.2 约束融合策略

规划Agent在生成方案时，需要同时满足时间约束、预算约束、偏好约束和地理约束。系统采用分层融合策略：先保证硬约束（日期、闭馆时间、预算上限），再优化软约束（兴趣匹配、体验多样性、路线舒适度）。当软约束冲突时，以用户显式偏好优先。

### 3.4.3 结果校验与修复策略

校验机制包含结构校验与业务校验。结构校验关注字段完整性与类型一致性，业务校验关注逻辑合理性（如时间顺序、预算累加、地点可达性）。若任一校验失败，系统触发修复提示词进行定向改写，最多重试固定次数。若仍失败，则返回可用的基础模板并提示用户补充信息。

## 3.5 系统可扩展性与演进设计

在毕业设计阶段，系统首先需要满足“可运行、可验证”的目标，但为了避免形成一次性工程，本文在设计阶段同步考虑可扩展性。可扩展性主要体现在功能扩展、数据扩展、模型扩展和部署扩展四个方面。

功能扩展方面，系统采用模块化服务设计。用户认证、行程规划、收藏管理、后台管理等模块以接口契约解耦，新增功能（如结伴出行、路线对比、费用分摊）可在不破坏既有模块的前提下独立演进。智能体层通过节点化编排支持新增角色Agent，例如“餐饮推荐Agent”“预算优化Agent”“交通拥堵规避Agent”等，扩展成本较低。

数据扩展方面，数据库设计遵循“核心稳定、外围可增”的原则。核心表结构保持主业务闭环，扩展表用于承载可选能力，如用户画像、行为埋点、策略评估结果等。通过外键和版本字段可实现平滑迁移，避免因字段激增导致核心链路性能下降。对于高频读场景，后续可引入缓存层与只读副本，进一步提升查询性能。

模型扩展方面，系统将模型调用抽象为统一适配层，不直接把业务逻辑绑定到单一模型供应商。未来可根据成本、性能与准确性在不同模型之间动态切换，或采用“小模型快速筛选+大模型精细生成”的混合策略。该设计能够降低平台锁定风险，增强系统生命周期内的技术弹性。

部署扩展方面，系统支持从单机部署演进到容器化部署。当前开发环境采用本地服务编排，后续可迁移到容器编排平台，实现弹性伸缩、灰度发布与滚动升级。通过服务健康检查、配置中心和日志聚合机制，系统可逐步向工程化运维体系过渡。

## 3.6 安全性、稳定性与工程规范设计

### 3.6.1 安全设计细化

安全设计贯穿输入、传输、存储与操作全过程。输入侧通过Pydantic模型严格限制字段类型与长度，防止异常输入触发系统错误；传输侧采用HTTPS与鉴权头规范，避免敏感信息明文泄漏；存储侧对密码等敏感数据进行不可逆摘要处理，不在数据库中保存明文凭据；操作侧通过角色权限与接口白名单限制高风险行为。

针对常见安全风险，系统采取了相应防护措施。对暴力登录尝试，采用失败次数统计与临时限制策略；对越权访问，通过中间件统一鉴权并在服务层二次校验；对管理接口，启用更严格的审计记录，确保每次关键操作可追踪、可回放、可归责。对于第三方工具调用，系统避免在前端暴露密钥，所有敏感调用由后端代理执行。

### 3.6.2 稳定性设计细化

稳定性设计目标是“局部故障不导致全局不可用”。系统在智能规划链路中引入超时控制、重试机制、降级返回和错误分类。对于短暂网络抖动引发的失败，优先重试；对于持续性服务不可达，返回可解释的降级方案；对于不可恢复错误，记录详细上下文并反馈明确提示，避免用户无感失败。

任务状态管理是稳定性的关键。系统将任务状态定义为“待执行、执行中、成功、失败、部分成功”五类，并在每次状态变化时写入日志。前端基于任务状态轮询展示进度，用户可感知系统仍在工作，降低误操作概率。该机制在长耗时任务场景下显著提升了交互体验。

### 3.6.3 工程规范与可维护性设计

系统实现遵循统一编码规范与目录规范。后端按“router-service-repository-schema”分层组织，前端按“page-component-store-api”组织；接口命名采用语义化路径与统一HTTP方法；错误码采用集中定义，避免不同模块同义不同码。文档方面，接口文档、数据库字典和部署说明保持同步更新，降低团队协作成本。

可测试性方面，系统把关键业务逻辑从控制器中剥离，便于单元测试与集成测试。对规划流程中的核心函数（输入标准化、结果校验、预算统计）优先补充测试样例。对于外部依赖，采用Mock方式构造稳定测试环境，减少不确定性。通过“可读日志+可测函数+可复现实验”，系统维护成本显著降低。

### 3.6.4 用户体验约束设计

智能系统若只追求技术指标而忽视用户体验，实际价值会大幅下降。本文在设计阶段加入了体验约束：  
（1）结果可解释：规划结果需包含理由描述与关键依据，帮助用户理解建议来源；  
（2）可编辑性：用户可手动调整行程而非被系统“锁死”；  
（3）可恢复性：误删、误改操作应提供提示和恢复路径；  
（4）可预期性：加载、失败、重试状态需可视化展示，避免“黑盒感”。  

这些约束虽然不直接提升算法分数，但显著提升了系统可用性与用户信任度，是毕业设计从“技术演示”走向“可用产品”所必须考虑的工程要素。

## 3.7 本章小结

本章在完成架构、数据库、模块和协同策略设计基础上，进一步补充了可扩展性、安全性、稳定性和工程规范设计。通过这些设计约束，系统不仅能够完成当前功能目标，还具备后续迭代的结构基础。整体上，本文形成了“业务建模—智能编排—工程治理”三位一体的设计框架，为后续实现与测试提供了充分支撑。

## 4.5 扩展测试设计与实验组织

为使测试结果更具说服力，本文在基础功能测试之外，补充了对比实验、压力场景实验和稳定性回归实验。扩展测试遵循“同条件、可重复、可解释”的原则：同一组输入在单智能体方案与多智能体方案下分别执行，记录完整日志并进行横向比较；对关键接口在不同并发等级下进行压测，分析吞吐与延迟变化；对典型异常输入进行重复执行，统计修复机制成功率和失败类型分布。

在实验组织上，本文将测试划分为离线测试与在线模拟两类。离线测试主要关注数据模型校验、规划结果解析、提示词模板稳定性等静态能力；在线模拟测试主要关注用户交互链路、外部工具调用稳定性、任务状态回写与页面展示一致性等动态能力。通过分层测试可避免“只测接口不测业务闭环”或“只测页面不测后端逻辑”的片面问题。

为保证实验结果客观，本文预先定义评估维度：  
（1）可用性维度：请求是否成功返回、是否能被前端完整展示；  
（2）正确性维度：关键字段是否完整、时间顺序是否合理、预算是否越界；  
（3）鲁棒性维度：异常输入、外部超时、部分数据缺失时系统是否可恢复；  
（4）效率维度：平均响应时间、任务完成时间、重试次数；  
（5）可维护性维度：错误信息是否可定位、日志是否可追踪、问题修复成本。  

在样本设计方面，本文设置了不同复杂度的测试样本：简单样本（单城市、短时长、约束少）、中等样本（多偏好、预算限制、需要权衡）、复杂样本（多天、多约束冲突、外部数据波动）。这种分层样本能更好呈现系统能力边界，避免仅凭少量成功案例得出过度乐观结论。

## 4.6 对比实验与结果讨论

### 4.6.1 多智能体方案与单智能体方案对比

为了验证多智能体协同的实际价值，本文构建了单智能体基线方案：由一个通用Agent同时完成信息检索、路线规划与结果输出。对比实验采用同样输入与同样外部工具，比较两种方案在结果质量、失败率和可解释性方面的差异。

实验观察表明，单智能体方案在简单场景下可以生成可读文本，但在复杂约束场景下更容易出现以下问题：  
第一，输出结构不稳定，字段缺失概率高；  
第二，预算估算与行程安排存在冲突，如“高成本酒店+低预算”并存；  
第三，路线连续性较弱，跨区域跳转频繁；  
第四，异常情况下难以修复，往往直接返回不可解析文本。  

多智能体方案通过角色分工显著缓解了上述问题。信息采集Agent只关注事实数据质量，规划Agent专注行程编排，校验Agent负责格式与逻辑检查，形成了“采集—生成—验证”闭环。虽然在部分场景下多智能体总体耗时略高于单智能体，但其成功率、可解析率和结果一致性明显更优，符合实际产品对稳定性的优先要求。

### 4.6.2 约束满足能力分析

旅行规划本质是多约束决策问题。本文重点分析以下约束满足能力：时间约束、预算约束、偏好约束和可达性约束。测试显示，系统在时间约束和预算约束上的满足率较高，主要得益于结构化校验和显式规则检查；在偏好多样性方面，系统能够根据兴趣标签调整景点类型，但在“偏好冲突”场景下仍有优化空间；在可达性方面，当前主要依赖地理位置与基础路线信息，尚未充分引入实时交通拥堵因子。

针对约束冲突场景，系统采用“硬约束优先、软约束优化”策略。硬约束包括日期合法、预算上限与基本可达性，软约束包括体验丰富度、节奏舒适度、主题连贯性。该策略可以保证结果“可执行”，但在“最优性”上仍可提升，例如可引入更细粒度评分函数对候选方案进行排序。

### 4.6.3 失败样本复盘分析

为避免“只报告成功案例”，本文对失败样本进行专项复盘。失败主要来源于三类：  
（1）外部工具异常：接口超时、临时限流、返回字段缺失；  
（2）输入约束冲突：预算过低同时要求高标准住宿与多景点覆盖；  
（3）模型输出偏差：部分字段缺失或单位不一致。  

针对上述失败，系统分别采用重试机制、冲突提示与降级模板。复盘结果显示，重试机制可覆盖大部分临时性外部异常；冲突提示可帮助用户快速调整输入；降级模板可避免“空白返回”，提升用户可接受度。该结果说明在智能应用中，异常处理与失败可解释机制与算法本身同等重要。

## 4.8 工程反思与改进方向

通过实现与测试过程，本文形成以下工程反思：

第一，智能应用的核心不仅是“模型效果”，更是“系统工程能力”。如果缺少明确的数据模型、错误码体系和日志追踪机制，即使模型输出看起来准确，也难以稳定服务真实用户。  
第二，多智能体协同并非越复杂越好，角色划分应以业务边界为依据。过度细分会引入额外通信成本，过度合并会导致职责混乱。  
第三，旅行规划属于高约束场景，必须把“可执行性”放在“文案华丽度”之前。结构化校验、业务规则检查和降级策略是必要而非可选项。  
第四，测试阶段应重视失败样本。失败样本比成功样本更能揭示系统短板，也是后续优化最有效的输入来源。  

面向后续迭代，本文建议从以下方向深化：  
（1）引入多目标优化算法，对成本、时间、体验和风险进行联合评分；  
（2）构建用户长期偏好记忆，支持跨会话个性化规划；  
（3）增加实时重规划能力，在天气突变或行程延误时自动给出替代方案；  
（4）建立更系统的评估基准集，持续跟踪不同版本的质量变化；  
（5）完善自动化测试流水线，提高回归效率与发布可靠性。

本文仍存在一定局限：第一，当前规划策略对多目标优化支持有限，在预算、时间、体验多样性冲突时尚未形成最优求解；第二，外部接口波动仍会影响部分数据完整度；第三，个性化记忆机制尚处于基础阶段，长期用户行为学习能力不足。未来工作可围绕多目标优化算法、用户画像持续学习、动态重规划机制和在线评估体系展开，进一步提升系统智能化水平与实际应用价值。

此外，后续研究还可引入更严格的离线评测基准与在线A/B评估机制，对不同模型版本、不同提示策略和不同协同拓扑进行长期跟踪比较，以形成量化的系统演进依据。若条件允许，可进一步接入真实出行反馈数据，把“用户满意度”“行程执行偏差率”“重规划触发率”等指标纳入统一评价框架，从而实现从原型系统到实用系统的持续优化闭环。
同时建议在课程设计、实验教学和校内项目孵化中复用本课题的工程模板，以持续验证其通用性和可迁移性。
